# v0.4.5

## Summary

This release replaces the one-shot config update flag with a versioned,
transactional external config upgrade system.

- binaries declare the required config schema and package ID
- users who skipped earlier config-changing releases are upgraded directly to
  the latest maintained config structure
- `config_up.zip` remains external and is verified through its manifest and
  per-file SHA-256 values
- failed or interrupted config updates are rolled back automatically

## Config update behavior

This release requires config schema `2`.

On first start after the binary update, MosDNS:

1. reads `webinfo/config_update_state.json`
2. downloads the external `config_up.zip`
3. verifies its channel, package ID, schema, file list, and hashes
4. creates a unique backup under `backup/`
5. updates only maintained structure files
6. creates missing switch state files without overwriting existing choices
7. validates the complete config tree and initializes plugins
8. commits schema `2` only after startup succeeds

If validation or startup fails, the original files are restored and the error
is exposed in the WebUI update panel.

## Safety boundaries

The package may replace only:

- `config_custom.yaml`
- maintained `sub_config/*.yaml` files

It does not overwrite user rules, upstream settings, appearance state,
dedicated group state, SRS data, generated files, or caches.

`rule/switch16.txt` is created with default value `B` only when it is missing.

## UI

The maintained system settings page now displays:

- required and applied config schema
- config update status
- config update error details when rollback occurs
