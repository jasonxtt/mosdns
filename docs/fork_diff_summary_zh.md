# 相对上游的改动说明

上游基础：

- `yyysuo/mosdns`

这个 fork 的方向不是重写上游核心，而是在尽量保持上游结构和配置思路的前提下，补齐更适合长期在线维护 DNS 分流规则的 WebUI 工作流。

## 一、专属分流组是什么

“专属分流组”可以理解为一组独立的域名分流槽位。命中该组的域名，会优先走：

- 该组绑定的专属上游
- 该组绑定的专属缓存
- 该组对应的专属规则链路

当前实现里，WebUI 管理的专属分流组最多支持 10 个，底层对应配置槽位：

- `special_50` 到 `special_59`

每个槽位会落到真实运行配置中的一整套对象：

- 专属规则集
- 专属上游
- 专属缓存
- 专属查询序列

## 二、如何在 UI 中添加和使用专属分流组

当前 WebUI 中的基本操作路径是：

1. 进入 `上游设置`
2. 点击 `新增专属分流组`
3. 输入组名，例如 `cmcc`
4. 在同一页面为这个组配置专属上游
5. 再到 `规则管理` 中把域名归入这个组

归入方式有两种：

- `本地规则`
  - 直接手工录入 `full: / domain: / keyword: / regexp:` 规则
- `订阅规则`
  - 添加在线规则源，让规则自动下载并更新

命中该组的域名会走它绑定的专属上游与缓存。

## 三、当前这条 fork 相对上游主要增强了什么

### 1. WebUI 专属分流组能力

- 新增专属分流组 API 和前端管理能力
- 可以把规则直接归入某个专属分流组
- 查询日志里按组名显示，减少内部 tag/mark 的阅读成本

相关接口：

- `GET /api/v1/special-groups/`
- `POST /api/v1/special-groups/`
- `DELETE /api/v1/special-groups/{slot}`

### 2. 规则维护工作流增强

- 在线分流规则保存后自动下载 `.srs`
- 新增规则时可根据 URL 自动识别名称和本地文件路径
- 本地规则、订阅规则、广告拦截的管理方式更统一

这部分主要解决的是：

- 规则来源维护
- 本地规则录入
- 新规则保存后还要手工更新的额外步骤

### 3. 上游管理增强

- WebUI 保存上游配置后支持热重载
- 专属分流组和专属上游直接联动
- 上游列表支持排序、隐藏未启用项等维护能力

核心效果是：

- 调整上游后可以即时生效
- 减少频繁重启 `mosdns` 带来的中断

### 4. 查询日志与排障增强

- 日志中显示更友好的专属分流组名称
- 补充匹配来源、最终上游组、最终上游
- 更适合排查“某个域名最终到底走了哪条规则和哪一路上游”

### 5. WebUI 主界面重构

- 合并原先分散的多个管理页面
- 重组为更清晰的主界面结构
- 将高频维护项从混杂布局中拆出来
- 统一确认交互与日常操作体验

## 四、相对上游的删减项

这个 fork 不是对上游所有能力面的全量继承。为了保持当前产品面清晰，已经明确收缩了与 WebUI 分流维护目标不一致的部分，尤其是：

- `nft` 相关能力
- `eBPF` 相关能力

这意味着：

- 上游在 `nft / eBPF / 系统劫持` 方向上的更新，不会默认同步到这个 fork
- 当前维护重点仍然是 WebUI、规则维护、上游管理和查询日志排障

## 五、这个 fork 的定位边界

这个 fork 不是单纯的上游镜像，也不是完全独立的新产品。它更像是一个围绕实际使用场景持续演进的 WebUI 增强维护分支，重点解决：

- 分流规则长期维护
- 上游在线管理
- 查询日志排障
- 家庭网络与代理分流场景下的运维效率

因此对外描述时，建议始终强调三点：

- 基于 `yyysuo/mosdns`
- 保持与上游一致的整体兼容思路
- 当前重点在 WebUI 和配置维护工作流增强
